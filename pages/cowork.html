The goal

Build a software-development pipeline that operates without human-in-the-loop intervention — agents take a feature ticket, write the spec, build the code, review their own work, run tests, evaluate user-visible quality, and ship a PR. The human role becomes architect and product owner, not implementer.

Why this is harder than “AI-assisted coding”
Engineering scope
  • Decide what to work on — sprint planning with dependency awareness, not just “next ticket.”
  • Coordinate multiple agents without lane collisions or duplicate work.
  • Detect its own failures and recover — loop guard, auto-recover dirty trees, mid-rebase detection.
  • Make merge / no-merge decisions on its own work without rubber-stamping.
  • Surface the right things to human attention — escalation triage, audit log, attention chips.
Where humans still belong
Honest scope
  • Architectural decisions — the shape of the system, not its tickets.
  • Production design — what the game should be, what good feels like.
  • Kill-switch decisions — emergency-stop, daily caps, blocked-path allowlists.
  • Writing the agent prompts and the constraints that make autonomy safe.
  • Reviewing the audit trail — the system shows its work; the human still reads it.

Architecture

Six subsystems — specialist roster, parallel orchestration, autonomous merge gate, self-healing knowledge loop, control dashboard, and defense-in-depth security — designed so each can fail without taking the others down.

1. Multi-agent specialist roster (16 agents)
Role-specialized prompts, strict output formats
  • Producer — owns the routing chain, dispatches juniors and reviewers, defends scope.
  • Story-designer / Game-balance / Art-director — write specs (Mechanics / Numbers / Visuals / Acceptance).
  • Junior-systems / Junior-content — implement code and data per spec.
  • Senior-reviewer / Security-reviewer / Refactor-cop — code review at architecture, security, and drift levels.
  • Bug-tester / Adversarial-tester — happy path plus break attempts.
  • Playtest-UX + 3 persona testers — speedrunner, completionist, casual.
  • Vision-keeper — final ship/hold gate.
  • Sprint-planner and Escalation-triager — daily lane allocation by domain; classifies stuck tickets into 5 actionable buckets.
2. Five-lane parallel orchestration
launchd · 30-minute tick · domain-allocated
  • Each lane is a launchd-scheduled bash script with its own log directory, lock file, and domain assignment (Combat, Content, Run/World, Visuals/UX, catch-all).
  • Lanes coordinate through a sprint plan in docs/sprint-plan.md that allocates tickets to lanes by domain.
  • Claim files prevent duplicate work; strict-mode picker idles a lane if its sprint section is empty (no cross-lane theft).
  • Cross-lane backfill is opt-in fallback when a lane is starved.
  • Health-based throttling auto-pauses lanes when pipeline completion drops below 60%.
3. Auto-gate (autonomous PR merge)
3-agent parallel voting · risk-bounded
  • When a junior opens a PR, three reviewer agents (senior + bug-tester + vision-keeper) evaluate in parallel and vote.
  • Auto-merges only when all three approve and risk gates pass.
  • Blocks on PR size > 1500 lines, blocked paths (orchestrator / agent / dashboard files), failing CI.
  • Caps daily auto-merges (configurable, defaults conservative).
  • Emergency-stops if 2+ decisions are reverted by a human within 24h.
  • Logs every decision to ~/.cowork-dashboard/auto-decisions/ with full reviewer rationale.
4. Self-healing knowledge system
JSONL events · pattern detection · 2-of-3 review gate
  • Every tick emits structured events to ~/.cowork-dashboard/events/<date>.jsonl.
  • A pattern detector identifies recurring failure modes via heuristic analysis.
  • A synthesis agent updates docs/learnings.md with patterns, workarounds, and anti-patterns.
  • A prompt-evolution loop proposes agent-prompt updates based on the learnings.
  • A 2-of-3 review gate (pattern-reviewer + vision-keeper) auto-applies safe updates, capped at 3 per day.
5. The Cowork Dashboard
Single-file Node + vanilla JS · mobile-first
  • Overview: hero card with phase progress, ETA + confidence interval, 14-day burn-down, attention chips for waiting work.
  • Sprint board: Azure-DevOps-style swimlanes per lane, status columns, drag-drop with kanban auto-commit, side drawer with full ticket details, WIP limits, configurable filters.
  • Analytics: KPIs, agent invocation breakdown, autonomous-decision audit log.
  • Audit: pattern detector, pipeline health, self-healing panel, prompt evolution, learnings render, security audit.
  • Chat: 4 context modes (Project chat, Project edit, Dashboard self-edit, Custom path), background-job streaming, retry-in-place, Free-CLI button to pause lanes during chat contention.
  • Help: ? button on every page with contextual walkthroughs.
  • Mobile-first: works at 390px without horizontal scroll, Tailscale-accessible from iPhone.
6. Defense-in-depth security
Auth · allowlists · rate limits · redaction
  • Bearer auth token on every elevated endpoint.
  • Path allowlist for chat custom-path mode (only ~/Projects, ~/dev, etc.).
  • 12 forbidden path patterns (.ssh, .aws, shell init files, SSH keys).
  • Safety preamble injected into every chat prompt.
  • Hard rejection of dangerous shell patterns (curl|bash, rm -rf, sudo, eval).
  • Magic-comment script signing — only # COWORK-SAFE scripts run via the dashboard.
  • Sliding-window rate limiting per token (30 elevated chats/hr, 20 script runs/hr).
  • Persistent audit log of every elevated operation.
  • Public demo mode with redaction layer (paths → ~/, emails → <email>, tokens → <redacted>) and 19 path-prefix blocks for sensitive endpoints.

Outcomes

Numbers reported the way the rest of the portfolio reports them — what was measured, how, and what is intentionally not claimed.

Capacity utilization recovered
Methodology
  • What: share of the 24-hour window in which the pipeline was advancing tickets vs. idling on stuck state.
  • How measured: empirical audit of structured events in ~/.cowork-dashboard/events/*.jsonl over a fixed window.
  • Result: 9.2% baseline → 30%+ after 8 ranked findings shipped (sprint-plan parser, push-retry consistency, loop-guard persistence, transient-state hygiene, Unity .meta workflow, etc.).
  • Not claimed: code quality improvement, ticket throughput per dollar, parity with a human team — those are separate effects.
  • Caveats: single-machine pipeline; numbers are internal; recovery ceiling is bounded by API rate limits and lane count, not architecture.
Auto-gate revert rate
Methodology
  • What: share of auto-merged PRs reverted by a human within 24h.
  • How measured: persistent JSONL log under ~/.cowork-dashboard/auto-decisions/; reverts cross-referenced from git log.
  • Result: 0% across the conservative ramp window (initial daily cap of 2; designed to scale with proven trust).
  • Not claimed: bug-free output — only that no merge crossed the “had to be reverted” bar inside the cap.
  • Safety net: emergency-stop fires after 2 reverts in 24h, freezing further auto-merges until a human re-arms the gate.
Manual deploys eliminated
Operational discipline
  • Every change to orchestrator, agents, and dashboard ships through the same commit / review / merge flow the system uses for itself.
  • Dashboard self-edit mode treats the control surface as a first-class deliverable — same gates, same audit log.
  • Public demo runs at automation.byheir.com via Cloudflare Tunnel + redaction middleware so live state is exposed without compromising the local control instance.
Foundational bugs fixed via measurement
Audit-driven engineering
  • Loop guard — caught lanes spinning on the same ticket; bounded retries before escalation.
  • Sprint-plan parser — silent strict-mode degradation that idled lanes despite available work.
  • Push retry consistency — every git push uses -u origin <branch> with exponential backoff (2s/4s/8s/16s).
  • Transient-state hygiene — ticket-attempts file wipes that broke escalation triage.
  • Unity .meta workflow — agents kept stripping .meta sidecars on edits, breaking the editor.

What I learned

The interesting work was not getting agents to write code. It was the engineering around them that turned them into a system.

Multi-agent coordination is mostly a logging and recovery problem
  • The agents writing code is the easy part. Catching the cases where they don't, classifying why, and routing the right next action is the work.
  • Structured event streams + a pattern detector beat “ask the LLM what went wrong” for every recurring failure mode.
  • Recovery is a first-class feature: dirty-tree auto-recover, mid-rebase detection, lock-file forensics, transient-state hygiene.
Defense in depth is a UX feature, not just a security one
  • Path allowlists and rate limits also serve as guardrails against confused agents.
  • They limit blast radius of mistakes, which means I can experiment more aggressively.
  • The redaction layer made it safe to put a public demo on the open web — same instance the system actually runs on.
The dashboard is the system's specification
  • What the dashboard surfaces becomes what the system optimizes for.
  • Wiring the right metrics into Analytics turned debugging structural bugs (silent strict-mode degradation, ticket-attempts file wipes) from days into minutes.
  • If a behavior is invisible on the dashboard, expect it to drift.
Autonomy is about the constraints, not the model
  • The system is autonomous within constraints — daily caps, size limits, blocked-path allowlists, emergency-stop on consecutive reverts.
  • The constraints are the design surface. The agents are the easy part to swap.
  • Trust is earned by tightening the gate, watching the audit log, and only loosening when reverts stay flat.

Stack

Boring building blocks deliberately. The novelty is the system, not the dependencies.

Runtime & orchestration
  • Node.js + Express — single-file server (~3,500 LOC) for the dashboard and the elevated-action API.
  • Vanilla JS + Tailwind + Chart.js — frontend (~3,800 LOC), no framework, no build step.
  • Bash + Python — orchestration scripts, tick runners, audit tooling.
  • launchd — schedules each lane independently with its own logs and lock files.
  • Git + GitHub CLI — every change ships as a PR; auto-gate decisions logged alongside.
Networking, AI, and exposure
  • Cloudflare Tunnel — exposes the local instance at automation.byheir.com without opening ports.
  • Tailscale — private mobile access (the same dashboard at 390px from an iPhone).
  • Claude Code + OpenAI API — the agent runtime; prompts versioned in the orchestrator repo.
  • Redaction middleware — public demo mode rewrites paths, emails, and tokens before they leave the instance.
